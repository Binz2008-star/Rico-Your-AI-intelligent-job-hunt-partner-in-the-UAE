# CLAUDE-MD-SOTA — CLAUDE.md Best Practices (Seed)

State-of-the-art reference for writing and maintaining `CLAUDE.md` files.
This is the **seed** file: web-sourced content only (Tiers 1–3), git-tracked.
When present, an enriched variant (`CLAUDE-MD-SOTA.enriched.md`, gitignored)
merges local `/insights` behavioral data on top of this seed.

Generated by the `/refresh-guidelines` skill. All content is source-attributed
(see **Source Attribution**); nothing here is invented.

## Purpose & Scope

- Use this document when creating a new `CLAUDE.md`, auditing an existing one
  (`/refactor-claude-md`), or scaffolding scoped files (`/scaffold-claude-md`).
- `CLAUDE.md` is Claude Code's persistent project memory. It is loaded into
  context **every session**, so its size and quality directly affect every
  interaction. [T1-002]
- Prefer the **enriched** guidelines when writing CLAUDE.md files; fall back to
  this seed when the enriched file is unavailable.

## Part 1: Structural Standards

- **Keep it concise.** For every line, ask: *"Would removing this cause Claude
  to make mistakes?"* If not, cut it. Bloated CLAUDE.md files cause Claude to
  ignore your actual instructions. [T1-002]
- **Loaded every session** — only include things that apply broadly. For domain
  knowledge or workflows relevant only sometimes, use **skills** instead; Claude
  loads them on demand without bloating every conversation. [T1-002]
- **File hierarchy** — CLAUDE.md can live in several locations, each with a
  scope: [T1-002]
  - `~/.claude/CLAUDE.md` (home): applies to all Claude sessions.
  - `./CLAUDE.md` (project root): checked into git, shared with the team.
  - `./CLAUDE.local.md`: personal project notes; add to `.gitignore`.
  - Parent directories: useful for monorepos — `root/CLAUDE.md` and
    `root/foo/CLAUDE.md` are both pulled in automatically.
  - Child directories: pulled in on demand when Claude works in that subtree.
- **Use Markdown structure** — headings and bullet lists. Group related rules
  under descriptive section headings so Claude can locate them. [T1-002, T3-002]
- After editing, run `/context` to confirm Claude loaded the file and see what
  it costs. [T1-002]

## Part 2: Content Guidelines

**Include** (only if removing it would cause mistakes): [T1-002]

- Project-wide **code style** rules (e.g. "use ES modules, not CommonJS";
  "destructure imports when possible"). [T1-002]
- **Workflow** rules that apply broadly: "typecheck when done with a series of
  changes"; "prefer running single tests, not the whole suite, for
  performance." [T1-002]
- Stable **conventions** — e.g. API conventions: kebab-case URL paths,
  camelCase JSON properties, pagination on list endpoints, version in the URL
  path (`/v1/`, `/v2/`). [T1-002]

**Exclude / anti-patterns:**

- Anything that does not change Claude's behavior — the removal test fails. [T1-002]
- Situational domain knowledge or long procedures — move these to **skills**,
  loaded on demand. [T1-002]
- Secrets, tokens, or environment-specific credentials.
- Content duplicated from other loaded files — prefer an `@import` over a copy.

## Part 3: Integration Guidelines

- **Imports (`@path`)** — pull other files into memory instead of duplicating:
  `@docs/git-instructions.md`, `@~/.claude/my-project-instructions.md`. An
  import whose path resolves outside the working directory is *external*; the
  first time Claude Code sees external imports in a project it shows an approval
  dialog. User-scope files (`~/.claude/CLAUDE.md`, `~/.claude/rules/`) load
  without a dialog. [T1-001, T1-002]
- **Skills vs CLAUDE.md** — CLAUDE.md is for always-on, broadly-applicable
  rules. Skills (`.claude/skills/<name>/SKILL.md`) hold repeatable, on-demand
  workflows invoked directly; set `disable-model-invocation: true` for
  explicit-only commands. [T1-002]
- **Rules directory (`.claude/rules/`)** — path-scoped instructions. Rules
  without a `paths` field load unconditionally; rules with glob `paths` (e.g.
  `**/*.ts`) trigger only when Claude reads a matching file. Use this to keep
  broad rules out of every-session context. [T1-001]
- **`/init`** — bootstraps a CLAUDE.md and incorporates existing Cursor
  (`.cursor/rules/`, `.cursorrules`) and Copilot
  (`.github/copilot-instructions.md`) rules. [T1-001]
- **`AGENTS.md` interop** — a symlink or `@AGENTS.md` import lets one file serve
  multiple agents; use the `@AGENTS.md` import on Windows where symlinks need
  elevated privileges. [T1-001]
- **CI/CD** — Claude Code can run in CI (e.g. GitLab) triggered by `@claude`
  mentions; keep CLAUDE.md free of pipeline secrets and pass credentials via CI
  variables / OIDC, not the memory file. [T1-004]

## Part 4: Behavioral Patterns

- State **workflow expectations** as imperative rules Claude can follow every
  time (typecheck after changes; run focused tests first). [T1-002]
- Encode **interaction preferences** the same way (plan-mode for sensitive
  paths, e.g. "use plan mode for changes under `src/billing/`"). [T1-001]
- Keep behavioral rules **testable and specific** — vague guidance ("write good
  code") wastes context; concrete rules ("version APIs in the URL path")
  change behavior. [T1-002, T3-001]
- Prefer **positive, actionable** phrasing over long prohibitions.

## Part 5: Maintenance Guidelines

- **Review cadence** — revisit CLAUDE.md when instructions are being ignored
  (often a bloat signal) or when conventions change; re-run the removal test on
  each line. [T1-002]
- **Self-improving loop** — when Claude repeats a mistake, add a single focused
  rule rather than a paragraph; when a rule stops mattering, delete it. [T1-002]
- **Verify after edits** — `/context` confirms the file is loaded and shows its
  token cost so growth stays visible. [T1-002]
- **Scope, don't centralize** — push situational detail into scoped child
  CLAUDE.md files, skills, or `.claude/rules/` instead of growing the root
  file. [T1-001, T1-002]
