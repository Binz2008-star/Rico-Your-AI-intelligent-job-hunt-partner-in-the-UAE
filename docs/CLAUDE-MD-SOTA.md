# CLAUDE.md Generation Guidelines (CLAUDE-MD-SOTA)

> Generated by `/refresh-guidelines` (claude-md-best-practices plugin). This is a **sub-product**: a reference for *how to write* CLAUDE.md files. It must never contain hyperlinks or references to internal dev docs — only to other sub-products or the external sources listed below.

## Purpose & Scope

This document is the seed reference (`docs/CLAUDE-MD-SOTA.md`, git-tracked, web sources only) consulted by `/refactor-claude-md` and `/scaffold-claude-md` when auditing or generating CLAUDE.md files. When a local `/insights` report is available, `docs/CLAUDE-MD-SOTA.enriched.md` merges this seed with behavioral data from actual usage; the enriched file takes precedence when present. Re-run `/refresh-guidelines` periodically to keep this current.

---

## Part 1: Structural Standards

- **Three-level hierarchy**: `~/.claude/CLAUDE.md` (global — personal preferences applying to every project), project root `CLAUDE.md` (team-shared, git-tracked), `CLAUDE.local.md` (personal overrides, gitignored). Root CLAUDE.md may also live at `.claude/CLAUDE.md`.
- **Directory-walk loading**: Claude Code loads CLAUDE.md/`CLAUDE.local.md` by walking up from the working directory. All discovered files concatenate (they do not override each other), ordered root-to-cwd — instructions closer to the launch directory are read last. Within a directory, `CLAUDE.local.md` is appended after `CLAUDE.md`.
- **On-demand subdirectory files**: CLAUDE.md files in subdirectories under the cwd are not loaded at launch — they load when Claude reads a file in that subdirectory. Use this for module-specific context (e.g. `src/auth/CLAUDE.md`) once the root file is pushing past its budget.
- **`claudeMdExcludes`**: in large monorepos, use this to skip other teams' CLAUDE.md files being picked up unintentionally.
- **`@path/to/file` imports**: embed another file's content at that point. Imports resolving outside the working directory ("external imports") trigger a one-time approval dialog the first time Claude Code encounters them in a project; user-scope imports (`~/.claude/...`) load without a dialog since the user wrote them.
- **AGENTS.md interop**: Claude Code reads `CLAUDE.md`, not `AGENTS.md`. If a repo already has an `AGENTS.md` for other agent tools, import it (`@AGENTS.md`) or symlink (`ln -s AGENTS.md CLAUDE.md`, not available on Windows without Developer Mode/admin) rather than duplicating content.
- **Size budget**: keep the root file **under 300 lines**; community consensus favors far shorter — HumanLayer's production root file is under 60 lines, and instruction-following degrades once a file drifts past ~80 lines. **Scoped/subdirectory files: budget ~60 lines** (smaller than root; they cover one directory's concerns only).
- **Formatting**: prefer bullet points and specific, actionable phrasing over prose. Verify Claude actually loaded the file with `/context` after edits.

## Part 2: Content Guidelines

**Include**: Bash commands Claude can't guess; code style rules that differ from language defaults; testing instructions and preferred test runners; repository etiquette (branch naming, PR/merge conventions); architectural decisions specific to the project; developer environment quirks (required env vars, toolchain versions); common gotchas or non-obvious behaviors.

**Exclude**: anything Claude can figure out by reading the code; standard language conventions Claude already knows; detailed API documentation (link to it instead); information that changes frequently; long explanations/tutorials; file-by-file codebase descriptions; self-evident advice ("write clean code").

For every line, ask: *"Would removing this cause Claude to make a mistake?"* If not, cut it. A bloated CLAUDE.md causes Claude to ignore the instructions that actually matter.

**Anti-patterns**:
- **Personality/tone instructions** ("be a senior engineer", "think step by step") — wastes tokens; the harness system prompt already covers this.
- **`@`-embedding entire docs by default** — this loads the whole file into every session. Instead, *pitch* Claude on when to read it on demand: "For Stripe integration issues, see `docs/stripe-guide.md`."
- **Style rules with no formatter behind them** — if a linter/formatter is configured, enforce it via a hook, not prose; don't make Claude an expensive, slow linter. If no formatter exists, concrete rules ("2-space indentation") are appropriate.
- **Cross-scope duplication** — repeating the same rule in global, project, and local files wastes tokens for no benefit.
- **Never include secrets** — API keys, credentials, connection strings, or exploitable vulnerability detail. CLAUDE.md is effectively part of the system prompt and should be treated as shareable documentation.

**Progressive disclosure**: for larger projects, keep task-specific instructions (build steps, test conventions, schema docs, architecture deep-dives) in separate named files (e.g. `agent_docs/*.md`) and reference them from CLAUDE.md with a one-line description of when to read each — Claude decides whether to pull it in. Prefer pointers (`file:line`) to copied snippets, which go stale.

## Part 3: Integration Guidelines

- **Hooks vs. CLAUDE.md**: deterministic, repeatable checks (formatting, linting) belong in a `Stop`/`PostToolUse` hook that runs the tool and surfaces errors to Claude — not as a CLAUDE.md instruction asking Claude to self-police style.
- **Skills vs. CLAUDE.md**: CLAUDE.md loads on *every* session regardless of relevance. Skills (`.claude/skills/`) load on demand. Move occasional or domain-specific workflows (e.g. "how to fix a GitHub issue") into a skill/slash command instead of the always-loaded file.
- **Rules (`.claude/rules/`)**: YAML-frontmatter files scoped with a `paths` glob (e.g. `src/**/*.ts`) that activate only when Claude reads a matching file; rules with no `paths` field apply unconditionally. `.claude/rules/` supports symlinks for sharing rule sets across projects.
- **Subagents (`.claude/agents/`)**: specialized assistants with their own context and tool scope, useful for isolated tasks (security review, large-scale search) without cluttering the main conversation.
- **Plugins**: bundle skills, hooks, subagents, and MCP servers into one installable unit (`/plugin` to browse/install).
- **CI/CD**: CLAUDE.md is read during headless/CI runs (e.g. GitLab `@claude` triggers) the same way as interactive sessions — keep it focused, since CI time is billed compute.

## Part 4: Behavioral Patterns

- **Instruction budget**: frontier thinking models reliably follow roughly 150–200 instructions; smaller/non-thinking models follow markedly fewer, degrading exponentially rather than linearly as instruction count grows. The Claude Code system prompt itself already carries ~50 instructions before CLAUDE.md, rules, skills, or user messages are added — leave headroom.
- **Positional bias**: LLMs weight instructions at the very start and very end of a prompt more heavily than the middle — front-load what matters most.
- **Uniform degradation, not selective ignoring**: as instruction count grows, adherence decreases *across all instructions*, not just the newest/lowest ones — an overloaded CLAUDE.md doesn't just drop the last few rules, it weakens all of them.
- **Relevance filtering**: Claude Code wraps CLAUDE.md in a system-reminder telling Claude the content "may or may not be relevant" and to ignore it if not. The more non-universal content in the file, the more likely Claude discounts the whole thing, including instructions that did matter.
- **Claude is not a linter**: reserve CLAUDE.md for judgment calls and project knowledge, not mechanical style enforcement a deterministic tool already handles.
- **Verification discipline**: typecheck after a series of changes; run a single relevant test rather than the whole suite while iterating; confirm the file loaded with `/context`.
- **Don't "set and forget" `/init`**: auto-generated CLAUDE.md is a starting draft, not a finished artifact — CLAUDE.md is the highest-leverage file in the harness (a bad line here compounds into bad plans and bad code across every session), so hand-review and prune every line over time rather than leaving the auto-generated version in place indefinitely.
- **Context hygiene**: prefer `/clear` + a project-specific catch-up routine over `/compact` for session resets; for large tasks, dump progress to a scratch `.md`, `/clear`, then resume by pointing Claude at that file.

## Part 5: Maintenance Guidelines

- **Treat CLAUDE.md like code**: review it when Claude's behavior goes wrong, prune it on a regular cadence, and validate edits by observing whether behavior actually changes — not just by re-reading the prose.
- **Self-improving pattern**: after correcting Claude on something non-obvious, ask it to update CLAUDE.md so the same mistake isn't repeated. This is the highest-leverage maintenance habit — the file compounds in value over time.
- **Version control**: check CLAUDE.md into git so the whole team benefits and can contribute; keep `CLAUDE.local.md` gitignored for personal-only content.
- **Periodic re-audit**: re-run `/refresh-guidelines` to refresh this reference, then `/refactor-claude-md` to re-check drift between actual CLAUDE.md files and current guidance — especially after the project's stack, structure, or team conventions change materially.
